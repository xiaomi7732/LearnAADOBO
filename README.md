# Learn Azure AD On-Behalf-Of (OBO) Flow with .NET

This solution demonstrates the **AAD OBO flow** end-to-end with 3 projects and 3 Azure AD app registrations.

## Architecture

```
┌─────────────┐       token A        ┌─────────────────┐     OBO → token B     ┌────────────────┐
│  ClientApp   │ ──────────────────►  │  MiddleTierApi  │ ───────────────────►  │  DownstreamApi  │
│  (Console)   │  scope: middle-tier  │  (Web API)      │  scope: downstream    │  (Web API)      │
│              │                      │  Port 7213      │                       │  Port 7022      │
└─────────────┘                      └─────────────────┘                       └────────────────┘
      │                                      │                                        │
  Interactive                          OBO Exchange:                            Validates token B
  browser login                    token A → token B                          (issued for this API)
```

### How OBO Works
1. **User signs in** via the ClientApp (interactive browser popup)
2. ClientApp receives **Token A** scoped to `api://MiddleTierApi/access_as_user`
3. ClientApp calls **MiddleTierApi** with Token A in the `Authorization` header
4. MiddleTierApi sends Token A to Azure AD and says: *"exchange this for a new token scoped to DownstreamApi, on behalf of this user"*
5. Azure AD returns **Token B** scoped to `api://DownstreamApi/.default`
6. MiddleTierApi calls **DownstreamApi** with Token B
7. DownstreamApi validates Token B and returns data

## Step 1: Azure AD App Registrations

You need to register **3 apps** in [Azure Portal → Microsoft Entra ID → App registrations](https://portal.azure.com/#view/Microsoft_AAD_IAM/ActiveDirectoryMenuBlade/~/RegisteredApps).

### 1A. Register: DownstreamApi

1. Click **New registration**
2. Name: `LearnOBO-DownstreamApi`
3. Supported account types: **Single tenant**
4. Click **Register**
5. Go to **Expose an API**:
   - Click **Set** next to Application ID URI → accept `api://<client-id>` or customize
   - Click **Add a scope**:
     - Scope name: `access_as_user`
     - Who can consent: **Admins and users**
     - Admin consent display name: `Access DownstreamApi as user`
     - Click **Add scope**
6. Note down:
   - **Application (client) ID** → `YOUR_DOWNSTREAM_API_CLIENT_ID`
   - **Directory (tenant) ID** → `YOUR_TENANT_ID`

### 1B. Register: MiddleTierApi

1. Click **New registration**
2. Name: `LearnOBO-MiddleTierApi`
3. Supported account types: **Single tenant**
4. Click **Register**
5. Go to **Expose an API**:
   - Click **Set** next to Application ID URI → accept `api://<client-id>`
   - Click **Add a scope**:
     - Scope name: `access_as_user`
     - Who can consent: **Admins and users**
     - Admin consent display name: `Access MiddleTierApi as user`
     - Click **Add scope**
6. Go to **Certificates & secrets** → **Certificates** → **Upload certificate**:
   - Upload the `.cer` file generated by `New-DevCertificate.ps1` (see Step 3)
7. Note down:
   - **Application (client) ID** → `YOUR_MIDDLETIER_API_CLIENT_ID`

### 1C. Register: ClientApp

1. Click **New registration**
2. Name: `LearnOBO-ClientApp`
3. Supported account types: **Single tenant**
4. Redirect URI: Platform = **Mobile and desktop applications** → `http://localhost`
5. Click **Register**
6. Note down:
   - **Application (client) ID** → `YOUR_CLIENT_APP_CLIENT_ID`

### 1D. Authorize the Apps (choose one approach)

The OBO flow requires Azure AD to know that ClientApp is allowed to call MiddleTierApi, and that MiddleTierApi is allowed to call DownstreamApi. There are **three ways** to set this up, each suited to different scenarios:

#### Option A: Pre-authorized Applications (Recommended for this project)

The **API owner** declares which apps are trusted — no consent prompts, no admin involvement needed.

Configure in Azure Portal → App registrations → **Manifest**:

**MiddleTierApi manifest** — pre-authorize the ClientApp:
```json
"preAuthorizedApplications": [
    {
        "appId": "<YOUR_CLIENT_APP_CLIENT_ID>",
        "delegatedPermissionIds": [ "<access_as_user scope ID from MiddleTierApi>" ]
    }
]
```

**DownstreamApi manifest** — pre-authorize the MiddleTierApi:
```json
"preAuthorizedApplications": [
    {
        "appId": "<YOUR_MIDDLETIER_API_CLIENT_ID>",
        "delegatedPermissionIds": [ "<access_as_user scope ID from DownstreamApi>" ]
    }
]
```

> You can find each scope's ID in the manifest under `oauth2Permissions[].id`.

**When to use**: All apps are owned by the same team. The API owner controls who can call it.
Trust is declared at the API level, and callers don't need any special configuration.

#### Option B: API Permissions + Admin Consent

The **calling app** declares what it needs, and an **admin** approves it centrally.

Configure on the **calling** app:
1. **MiddleTierApi** → API permissions → Add `LearnOBO-DownstreamApi` → `access_as_user` → Grant admin consent
2. **ClientApp** → API permissions → Add `LearnOBO-MiddleTierApi` → `access_as_user` → Grant admin consent

**When to use**: Corporate/production environments where admins control which apps can access which APIs.
Common when apps are owned by different teams, or when user consent is disabled org-wide.

#### Option C: User Consent (Simplest, no configuration)

If the scopes are configured with **Who can consent: Admins and users**, then the user is prompted
to consent at sign-in time. No API permissions or pre-authorization configuration needed.

**When to use**: Development and personal tenants where you are the user and the admin.
Not suitable for production — offers no centralized control over app-to-app trust.

#### Summary

| Approach | Configured on | Who approves | Consent prompt | Best for |
|----------|--------------|--------------|----------------|----------|
| **Pre-authorized apps** | The API (receiver) | API owner | None | Same-team apps |
| **API permissions** | The client (caller) | Tenant admin | None (after admin grants) | Cross-team / corporate |
| **User consent** | Nothing | End user | Yes, at sign-in | Dev / personal tenants |

## Step 2: Configure the Projects

Replace the placeholder values in each project's config files:

### DownstreamApi/appsettings.json
```json
{
  "AzureAd": {
    "Instance": "https://login.microsoftonline.com/",
    "TenantId": "<YOUR_TENANT_ID>",
    "ClientId": "<YOUR_DOWNSTREAM_API_CLIENT_ID>",
    "Audience": "<YOUR_DOWNSTREAM_API_CLIENT_ID>"
  }
}
```

> **Note**: The `Audience` values above use the raw client ID GUID (v2 token format).
> This requires `"requestedAccessTokenVersion": 2` in each API's app manifest (see Common Issues).
> If using v1 tokens (the default), use `"api://<CLIENT_ID>"` instead.

### MiddleTierApi/appsettings.json
```json
{
  "AzureAd": {
    "Instance": "https://login.microsoftonline.com/",
    "TenantId": "<YOUR_TENANT_ID>",
    "ClientId": "<YOUR_MIDDLETIER_API_CLIENT_ID>",
    "CertificatePath": "MiddleTierApi.pfx",
    "CertificatePassword": "dev-only-password",
    "Audience": "<YOUR_MIDDLETIER_API_CLIENT_ID>"
  },
  "DownstreamApi": {
    "BaseUrl": "https://localhost:7022",
    "Scopes": [ "api://<YOUR_DOWNSTREAM_API_CLIENT_ID>/.default" ]
  }
}
```

### ClientApp/Program.cs (constants at the top)
```csharp
const string tenantId = "<YOUR_TENANT_ID>";
const string clientAppClientId = "<YOUR_CLIENT_APP_CLIENT_ID>";
const string middleTierApiScope = "api://<YOUR_MIDDLETIER_API_CLIENT_ID>/access_as_user";
```

## Step 3: Generate the MiddleTierApi Certificate

The MiddleTierApi uses a certificate (instead of a client secret) to prove its identity to Azure AD during the OBO flow. Run the included script to generate a dev certificate:

```powershell
.\New-DevCertificate.ps1
```

This creates two files in `MiddleTierApi/`:
- **`MiddleTierApi.pfx`** — private + public key, loaded by the code at runtime
- **`MiddleTierApi.cer`** — public key only, **upload this** to Azure Portal → App registrations → MiddleTierApi → Certificates & secrets → Certificates

Optional parameters:
```powershell
.\New-DevCertificate.ps1 -DaysValid 7              # shorter lifetime
.\New-DevCertificate.ps1 -PfxPassword "my-secret"   # custom password (update appsettings.json to match)
```

> ⚠️ If your org has a credential lifetime policy, you may need to reduce `-DaysValid` to comply.

## Step 4: Run

Open **3 terminals** in the solution root:

```bash
# Terminal 1 — Start DownstreamApi (port 7022)
dotnet run --project DownstreamApi --launch-profile https

# Terminal 2 — Start MiddleTierApi (port 7213)
dotnet run --project MiddleTierApi --launch-profile https

# Terminal 3 — Run ClientApp
dotnet run --project ClientApp
```

The ClientApp will:
1. Open your browser for interactive login
2. Acquire a token scoped to the MiddleTierApi
3. Call `GET /api/weather-aggregator` on the MiddleTierApi
4. The MiddleTierApi will perform OBO and call the DownstreamApi
5. You'll see the aggregated response in the console

## Common Issues

| Symptom | Fix |
|---------|-----|
| `IDX10214: Audience validation failed` | The token's `aud` claim format doesn't match the configured `Audience`. This is a **v1 vs v2 token** issue. By default, new app registrations issue v1 tokens where `aud` = `api://<client-id>`. To use v2 tokens (where `aud` = the raw client ID GUID), go to Azure Portal → App registrations → **Manifest**, and set `"requestedAccessTokenVersion": 2` under the `api` section. Do this for both **MiddleTierApi** and **DownstreamApi**. Then set `Audience` in `appsettings.json` to the raw client ID (no `api://` prefix). |
| `AADSTS65001: The user or administrator has not consented` | Go to API permissions → Grant admin consent |
| `AADSTS700027: Client assertion contains an invalid signature` | The certificate uploaded to the app registration doesn't match the `.pfx` used by the code. Re-upload the `.cer` file or regenerate the certificate. |
| `AADSTS7000215: Invalid client secret` | The client secret or certificate is not recognized. Check Certificates & secrets in Azure Portal. |
| `401 Unauthorized` on DownstreamApi | Check the `Audience` in appsettings matches the exposed API URI |
| `AADSTS50011: The redirect URI does not match` | Ensure ClientApp registration has `http://localhost` as redirect URI |
